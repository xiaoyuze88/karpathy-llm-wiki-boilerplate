# lint-wiki

扫描 `wiki/` 目录，检测质量问题，输出诊断报告。

## 执行步骤

1. **扫描所有 wiki/ 文件**
   - 读取 `wiki/index.md` 获取全局视图和页面列表
   - 逐一读取所有子目录下的 .md 文件

2. **检测矛盾**
   - 找出不同页面对同一概念的相互矛盾描述
   - 标记可能过时的具体数据/版本号（创建时间距今 >6 个月）

3. **找出孤立页面**
   - 没有被任何其他 wiki/ 页面以 `[[链接]]` 形式引用的页面

4. **找出缺失概念**
   - 在 ≥3 个页面中被提及但没有专属 `wiki/concepts/` 页面的词汇/概念

5. **检查结构完整性**
   - `wiki/index.md` 中的计数是否与实际文件数一致
   - 各页面 frontmatter 是否完整（title/created/modified/tags/summary 五个必填字段）
   - `wiki/log.md` 中的条目格式是否统一（应为 `## [YYYY-MM-DD] 操作类型 | 描述`）

6. **建议新来源**
   - 根据现有 Wiki 的主题范围和空白区域，推荐 5-10 个值得补充的来源方向
   - 每条建议说明：主题 + 为什么有价值 + 可能的搜索关键词

7. **生成报告并制定修复方案**（输出到对话）

   报告格式：
   ```
   ## Lint 报告 [{日期}]

   ### 矛盾 ({n} 处)
   - 页面A vs 页面B：{描述矛盾内容}
     → 建议：保留哪个版本，或如何调和

   ### 孤立页面 ({n} 个)
   - [[页面路径]] — {原因}
     → 建议：添加到哪个页面的引用中，或删除

   ### 缺失概念 ({n} 个)
   - "{概念词}" — 出现在：{页面列表}
     → 建议：新建 wiki/concepts/{slug}.md

   ### 结构问题 ({n} 处)
   - {具体问题描述}
     → 建议：{修复方法}

   ### 过时声明 ({n} 处)
   - 页面：{路径}，内容：{原文}，创建时间：{日期}
     → 建议：更新或标注过时

   ### 建议新来源 ({n} 个)
   - {主题}：{理由} [搜索词：{关键词}]
   ```

   报告生成后，按优先级列出修复计划：
   ```
   ## 修复计划

   P0 [立即修] {问题} → {具体操作}
   P1 [建议修] {问题} → {具体操作}
   P2 [可选修] {问题} → {具体操作}
   ```

   **输出修复计划后等待用户确认，用户同意后按优先级逐项执行修复（直接编辑 wiki 页面）。**

8. **执行修复（用户确认后）**
   - 按 P0 → P1 → P2 顺序逐项修复
   - 每修复一项向用户简报："已修复：{问题}"
   - 全部完成后进入下一步

9. **追加到 `wiki/log.md`**
   ```
   ## [{日期}] lint | 健康检查
   - 矛盾: {n} 处（已修 {m} 处）
   - 孤立页面: {n} 个（已修 {m} 个）
   - 缺失概念: {n} 个（已修 {m} 个）
   - 结构问题: {n} 处（已修 {m} 处）
   - 过时声明: {n} 处（已修 {m} 处）
   ```

10. **更新 `VAULT-INDEX.md`**
    - 最近活动追加本次 lint 记录

11. **更新 `wiki/hot.md`**
    - 在"最近活动"区块追加：
      ```
      - {日期} lint | 发现 {n} 个问题，已修复 {m} 个，待处理：{未修问题列表}
      ```
    - 如 hot.md 超过 500 字，压缩最早的活动记录

12. **Git 同步**
    ```bash
    # 如果知识库不在 ~/knowledge-vault，请将下面路径替换为实际路径
    cd ~/knowledge-vault
    git add .
    git commit -m "lint: 健康检查 {日期}"
    git push
    ```

## 完成后输出

Lint 报告 + 修复计划（等待用户确认）→ 执行修复 → 汇报修复结果 + Git push 状态。
